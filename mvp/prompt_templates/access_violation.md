## 分析重点: ACCESS_VIOLATION (C0000005)
本次崩溃是访问违例，常见根因:
1. **空指针解引用** (null pointer dereference) — 访问地址 0x00000000
2. **缓冲区溢出** (buffer overflow) — 覆盖返回地址或虚函数表
3. **use-after-free** — 已释放内存后被再次使用
4. **栈损坏** (stack corruption) — 返回地址被覆盖
5. **空对象成员访问** — 访问偏移量小的地址 (如 +0x20)
6. **内存双重释放** (double free) — 同一块内存释放两次
7. **野指针** (wild pointer) — 指针指向无效地址

### 2. 根因分析 — 重点排查:
- **访问地址分析**: 
  - 0x00000000 → 空指针
  - 小偏移 (如 0x00000020) → 空对象的成员
  - 看起来像已释放内存的模式 → use-after-free
  - 正常范围内的地址但无法访问 → 权限问题或已释放
- **寄存器分析**: 检查 rax/rcx/rbx 等通用寄存器是否持有非法地址
- **调用栈完整性**: 检查是否有被覆盖的返回地址 (栈损坏)
- **参数验证**: 崩溃函数的参数来源，是否可能为 NULL
- **多线程竞态**: 是否有其他线程可能同时释放/修改该内存

## 注意事项
- **★ 系统环境信息来自崩溃机器（DMP内部）**，不是分析者的电脑
- 访问地址为 0x00000000 时，确认哪个指针是 NULL，追溯其来源
- 访问地址为小值时，检查是否为通过空对象指针访问成员 (obj->member)
- 调用栈顶部异常且帧看起来损坏，检查是否栈缓冲区溢出
- 如果多个线程同时崩溃或持有锁，优先考虑竞态条件
- 结合堆状态 (!heap -s): 堆损坏时 use-after-free 可能性大
- 结合内存分析: 如果系统内存不足，可能是 OOM 导致分配返回 NULL
