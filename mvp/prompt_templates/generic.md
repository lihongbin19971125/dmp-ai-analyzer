## 分析要求 (必须逐项回答)

### 1. 崩溃摘要

### 2. 根因分析 (最重要)

要求:
- 每个结论**必须引用具体数据作为证据** (调用栈帧、寄存器值、内存状态等)
- 从以下维度逐一排查: 代码逻辑、内存管理、多线程/锁、系统资源、第三方库、环境配置
- 如果信息不足以确定根因，明确说明缺少什么信息以及可能的排查方向
- 区分"症状"和"根因"：例如空指针访问是症状，没有对 API 返回值做检查才是根因

### 3. 修复建议

- 如果没有源码，给出**排查和修复方向**

### 4. 置信度
- High: 有明确证据链支持，基本确定根因
- Medium: 有部分证据，但存在其他可能性
- Low: 信息不足，仅能推测

### 5. 预防措施
如何避免同类问题再次发生: 代码规范、静态分析、测试用例、监控告警等。

## 分析注意事项

- **★ 系统环境信息来自崩溃机器 (DMP 内部)，不是分析者的电脑**
- 结合系统内存状态判断: 是否 OOM、页面文件耗尽、虚拟地址空间不足
- 结合 CPU 特性判断: 是否指令集不兼容 (如 AVX-512 在旧 CPU 上)
- 结合系统运行时间: 是否长时间运行导致资源泄漏 (handle leak, memory leak)
- 注意多线程竞态: 检查是否有其他线程持锁或正在修改相关内存
- 注意堆状态: 堆损坏通常意味着 use-after-free 或 buffer overflow
- 如果日志中有错误/异常模式，结合调用栈交叉分析
- 如果近期有代码变更 (git log)，重点关注变更引入的问题
- 如果异常码是 C0000017 (STATUS_NO_MEMORY):
  - 系统内存不足 → 根因可能是 OOM 而非代码 bug
  - 系统内存充足 → 可能是申请了非法大小的内存 (如负数转 size_t)
- 如果异常码是 C0000005 (ACCESS_VIOLATION):
  - 访问地址为 0x00000000 → 典型的空指针解引用
  - 访问地址为小值 (如 0x20) → 可能是通过空对象偏移访问成员
  - 访问地址看起来像已释放的内存 → use-after-free
